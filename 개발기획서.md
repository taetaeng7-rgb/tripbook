## 0. Executive Summary

**한 문장 비전:** 도쿄에 사는 사람이 "이번 달(다음 달) 어디 가지?"를 근거와 함께 5분 안에 정하게 한다.

여행 계획의 3대 페인포인트를 해결한다.

1. **시즌 정보의 부재** — 같은 여행지도 시기에 따라 천국과 지옥이 갈린다(우기의 발리, 태풍철 오키나와, 한여름 교토). "어디가 좋다"는 정보는 넘치지만 "**언제** 가야 하는지"는 흩어져 있다.
2. **국내 vs 해외 비교 불가** — 같은 연휴를 놓고 일본 국내와 해외를 저울질할 때, 시기 적합성·이동시간·예산을 **같은 기준으로 나란히** 볼 도구가 없다.
3. **광고·잡음** — 검색 결과는 어필리에이트와 협찬 글 범벅. 추천에 신뢰할 수 있는 "이유"가 없다.

**핵심 차별점 3가지**
1. **월별 시즌 캘린더** — 1~12월 × 국내/해외 매트릭스. 매월 국내 3곳 + 해외 3곳을 큐레이션.
2. **근거 중심 추천** — 모든 추천에 "왜 이 달인가"를 5대 근거(기후 / 자연경관 / 축제·이벤트 / 비용·혼잡 / 액티비티)로 명시. "경치가 아름답다"가 아니라 "라벤더가 7월 중순 절정"이라고 쓴다.
3. **도쿄 출발 실용 정보** — 이동시간(하네다/나리타/도쿄역 기준), 예산 레벨(¥~¥¥¥¥¥), 추천 일수, 주의사항(태풍·혹서·혼잡)까지 한 카드에.

**기술 방향:** food-recipy에서 검증된 구조를 그대로 재사용 — 백엔드 없는 **바닐라 HTML+CSS+JS(ES 모듈) 정적 웹앱**. 빌드 없이 `git push`로 GitHub Pages 배포. 여행지는 **JSON 파일**로 관리해 코드 수정 없이 추가 가능. 이 앱의 복잡성은 UI가 아니라 **콘텐츠(시즌 근거의 정확성)**에 있으며, 이는 데이터 검증 스크립트와 연 1회 감수 운영으로 방어한다.

**MVP 범위:** 홈(이번 달 추천 + 다음 달 미리보기) → 월별 캘린더(1~12월) → 국내/해외 구분 브라우즈 → 여행지 상세("왜 지금인가" 중심) + 시드 여행지 **72곳(매월 국내 3 + 해외 3)**. "월을 고르면 갈 곳이 이유와 함께 나오고, 국내/해외를 나눠 볼 수 있으면" MVP 완료.

## 1. 종합 결정사항 (Single Source of Truth)

> 이후 모든 섹션이 따르는 **공통 기준**. food-recipy에서 검증된 결정은 재사용하고, 여행 도메인 고유 결정을 추가했다.

### 1.1 기술 스택
- **바닐라 HTML + CSS + JS(ES 모듈)**, 빌드 없음. → 근거·비교는 §5.1.
- **GitHub Pages** 배포. 자산·데이터는 **상대경로**, **해시 라우팅** 사용. **`.nojekyll` 파일 필수**(한글 파일명·`_` 경로 이슈 방지, food-recipy에서 학습한 사항).
- PWA(오프라인·홈 화면 추가)는 **Phase 3** 선택 기능.

### 1.2 데이터 저장
- **구분별 JSON 파일**: `data/destinations/국내.json`, `data/destinations/해외.json`. 각 파일 = 여행지 객체 배열. 추가 = 객체 하나 append.
- **월별 큐레이션 인덱스**: `data/calendar.json` — 매월 국내 3 + 해외 3의 여행지 id를 **편집 순서대로** 참조. (여행지의 `bestMonths`에서 자동 유도하지 않고 사람이 큐레이션 — "이 달의 대표 3곳"은 편집 판단이기 때문.)
- 파일명·URL에 한글 사용 가능(fetch 시 `encodeURIComponent`).

### 1.3 분류·용어 표준

**구분(scope)** — 이 앱의 제1 탐색축. 모든 화면에서 국내/해외를 구별해 보여준다.
| 값 | 의미 |
|---|---|
| `domestic` | 일본 국내 (거주지 기준) |
| `overseas` | 해외 (일본 출발 국제선) |

**국내 지방 구분(region, 9종)**: `홋카이도` `도호쿠` `간토` `주부` `간사이` `주고쿠` `시코쿠` `규슈` `오키나와`

**해외 권역 구분(region, 8종)**: `한국` `동아시아`(대만·홍콩·중국) `동남아시아` `남아시아·중앙아시아` `중동·아프리카` `유럽` `오세아니아·태평양` `아메리카`
> 한국을 별도 권역으로 두는 이유: 사용자에게 가장 빈번한 목적지(본가 방문 결합)이므로 브라우즈 최상단 고정.

**테마 태그(themes, 통제 어휘)**: `벚꽃` `단풍` `꽃` `설경` `스키` `온천` `바다·리조트` `축제` `트레킹` `자연` `미식` `도시` `문화·유적` `오로라` `사파리` `피서` `피한` `일루미네이션` `드라이브` `산악`

**시즌 근거 카테고리(reasonTags, 5종)** — "왜 이 달인가"의 근거 유형. 모든 월별 이유에 1개 이상 태깅.
| 카테고리 | 예 |
|---|---|
| `기후` | 건기/우기, 기온, 장마·태풍 회피 |
| `자연` | 개화·단풍·설경·유빙·백야 등 자연 현상의 절정 |
| `축제` | 마츠리·페스티벌·일루미네이션 등 이벤트 |
| `비용·혼잡` | 비수기 요금, 성수기 직전/직후의 한산함 |
| `액티비티` | 스키·다이빙·트레킹·등산 등 시즌 한정 활동 |

**예산 레벨(budgetLevel, 1~5)** — 1인 총액 대략(항공/교통+숙박+현지비), 표준 일수 기준.
| 레벨 | 표기 | 기준 | 예 |
|---|---|---|---|
| 1 | ¥ | ~3만엔 | 간토 근교 당일~1박 |
| 2 | ¥¥ | 3~7만엔 | 국내 주말 2박, 한국·대만 짧은 여행 |
| 3 | ¥¥¥ | 7~15만엔 | 홋카이도·오키나와 3박, 동남아 리조트 4~5일 |
| 4 | ¥¥¥¥ | 15~30만엔 | 유럽·북미·하와이 1주 |
| 5 | ¥¥¥¥¥ | 30만엔~ | 몰디브 수상빌라, 사파리, 남미·아프리카 장거리 |

**이동 기준**: 도쿄 출발(하네다/나리타 공항, 신칸센은 도쿄역). `access.hours`는 대략적인 편도 소요(환승 포함 체감치).

### 1.4 월별 추천 규칙
- `calendar.json`이 매월 **국내 3 + 해외 3**을 id로 큐레이션. 배열 순서 = 표시 순서(1번이 그 달의 대표).
- 각 항목에 `peak: true` 지정 가능 → "지금이 절정" 배지.
- 홈 화면은 **현재 월을 자동 감지**해 그 달의 추천을 표시하고, 아래에 **다음 달 미리보기**를 붙인다(여행은 선행 예약이 기본이므로).
- 12월의 다음 달 = 1월(순환).
- 월별 이유(`monthlyReasons`)는 반드시 **그 달만의 구체적 사실**을 담고 근거 카테고리를 태깅한다. 금지: "아름답다", "인기가 많다" 같은 무근거 서술.

### 1.6 열린 이슈 (개발 착수 전 확인)
- **홈의 기본 월**: 이번 달 vs 다음 달. → **이번 달 메인 + 다음 달 미리보기 섹션**으로 결정(즉흥 여행과 선행 계획을 모두 커버). 사용 데이터가 쌓이면 재검토.
- **사진**: 저작권 리스크 때문에 MVP는 **사진 없이** 테마 이모지 + 구분별 그라디언트 카드로 출발. 직접 촬영분/퍼블릭 도메인 확보 시 Phase 3에서 도입.
- **한/일 언어 토글**: 데이터는 `name.local` 등으로 이중화를 준비하되, UI 토글은 Phase 3(food-recipy와 동일 패턴 이식 가능).
- **환율·물가 변동**: `budgetLevel`은 대략 레벨이므로 연 1회 감수 시 함께 점검.

## 2. 제품 기획

### 2.1 제품 비전 & 문제 정의

**비전 한 문장:** 도쿄에 사는 사람이 "이번 달(다음 달) 어디 가지?"를 근거와 함께 5분 안에 정하게 한다.

도쿄 거주자가 여행을 계획할 때의 현실:

- **"언제"가 정리된 곳이 없다.** 여행지 정보는 블로그·유튜브·SNS에 넘치지만, 전부 "어디가 좋았다"는 후기다. 오키나와는 태풍철(8~9월)을 피해야 하고, 발리는 우기(12~2월)가 있고, 교토 단풍은 11월 중~하순 딱 2주다. 이 **시즌 적합성**을 매번 처음부터 검색해 조합해야 한다.
- **국내와 해외를 저울질할 도구가 없다.** "9월 실버위크에 어디 가지?"라는 질문에는 국내(다이세츠산 첫 단풍)와 해외(더위 꺾인 남유럽)가 모두 후보다. 그런데 국내 정보와 해외 정보는 서로 다른 사이트에 있고, 이동시간·예산 기준도 제각각이라 나란히 비교할 수 없다.
- **검색 결과를 신뢰할 수 없다.** 상위 노출 글은 어필리에이트 링크와 협찬이 섞여 있고, "무조건 가야 할 BEST 10" 류의 근거 없는 나열이 대부분이다.

이 앱은 **월(1~12) × 구분(국내/해외)** 매트릭스에 근거 있는 큐레이션을 채워 이 문제를 정면으로 해결한다.

### 2.2 타겟 사용자 & 사용 시나리오

**주 페르소나 — "도쿄 직장인 taemin"**
- 도쿄 거주 한국인 직장인. 주말·3연휴에는 국내(온천·자연·마츠리), 긴 연휴(GW·오봉·연말)에는 해외를 노린다.
- 한국 본가 방문과 여행을 결합하기도 한다(한국이 가장 빈번한 해외 목적지).
- 계획은 폰으로, 이동 중·자기 전에 세운다. 앱 설치 없이 링크로 바로 보고 싶다.
- "가서 뭐 하지"보다 "**언제 어디가 제철이지**"가 늘 첫 질문.

**대표 사용 상황**
1. **연휴 계획 (실버위크 2달 전):** 9월 연휴에 어디 가지? → 앱에서 9월을 열고 국내(다이세츠산 첫 단풍)와 해외(스페인, 터키)를 나란히 비교 → 이동시간·예산 보고 결정 → 항공권 예약.
2. **즉흥 주말 (목요일 밤):** 이번 주말 훌쩍 다녀올 데 없나? → 홈의 "이번 달" 국내 추천에서 이동 3h 이내·1박 가능한 곳을 고름 → 금요일에 숙소 예약.
3. **연간 계획 (연초):** 올해 연차를 어디에 쓸까? → 1~12월을 훑으며 "4월 알펜루트 설벽, 7월 몽골, 11월 교토 단풍"처럼 1년 여행 지도를 그림.

### 2.3 핵심 가치 제안

일반 여행 블로그·검색 대비 차별점 3가지.
1. **월별 시즌 캘린더:** "지금(다음 달) 가기 좋은 곳"이 항상 첫 화면. 1~12월 어느 달을 열어도 국내 3 + 해외 3이 이유와 함께 나온다.
2. **근거 중심:** 모든 추천에 5대 근거(기후/자연/축제/비용·혼잡/액티비티) 태그와 구체적 사실("수빙이 2월에 최대 성장", "11~2월이 태국의 시원한 건기"). 광고 없음.
3. **국내/해외 동일 기준 비교:** 구분은 확실히 나누되, 이동시간·예산 레벨·추천 일수라는 같은 잣대로 표기해 연휴 하나를 두고 국내와 해외를 저울질할 수 있다.

### 2.4 기능 목록 & MVP 범위

우선순위: **P0 = MVP 필수(첫 출시)**, **P1 = 다음 반복**, **P2 = 여유되면**.

| 기능 | 우선순위 | 근거 |
|---|---|---|
| 홈: 이번 달 추천(국내 3+해외 3) + 다음 달 미리보기 | P0 | 앱의 존재 이유 |
| 월별 캘린더 뷰(1~12월 이동, 국내/해외 세그먼트) | P0 | 제1 탐색축 |
| 국내/해외 구분 브라우즈(지방·권역별 그룹) | P0 | 사용자 명시 요구: 국내·국외 구별 |
| 여행지 상세: "왜 이 달인가" + 하이라이트 + 실용정보 | P0 | 근거 중심의 완결 |
| 시드 여행지 72곳(매월 국내 3+해외 3) | P0 | 12개월 전부 채워야 캘린더가 성립 |
| 모바일 우선 반응형 + 다크모드 | P0 | 폰으로 계획하는 게 전제 |

| 데이터 검증 스크립트(`validate:data`) | P0 | 콘텐츠 무결성은 이 앱의 생명 |
| 테마 필터(온천·바다·축제…) | P1 | 여행지 늘면 필요 |
| 검색(이름·국가·테마) | P1 | 〃 |
| 위시리스트·가봤음 체크(localStorage) | P1 | 재방문 가치 |
| 조건으로 찾기: 월 × 일수 × 테마 → 매칭 | P1 | food-recipy '냉장고 찾기'에 해당하는 킬러 편의 |
| 일본 연휴 캘린더 오버레이(GW·오봉 등 표시) | P1 | 연휴 계획 시나리오 강화 |

**MVP 정의:** P0만으로 "홈에서 이번 달 추천이 뜨고 → 월을 바꿔가며 1년치를 훑고 → 국내/해외를 나눠 보고 → 카드를 누르면 왜 지금인지와 실용정보가 나오는" 상태. 여행지는 JSON으로 관리해 코드 수정 없이 추가.

### 2.5 사용자 스토리
1. 도쿄 거주자로서, **앱을 열면 이번 달·다음 달에 가기 좋은 곳부터** 보고 싶다. 여행 생각은 늘 "지금 기준"으로 시작되기 때문이다.
2. 연휴를 앞둔 사람으로서, **국내와 해외 후보를 같은 화면에서 비교**하고 싶다. 사이트를 오가며 기준이 다른 정보를 짜맞추기 싫기 때문이다.
3. 계획러로서, **추천마다 "왜 이 달인가"라는 근거**를 보고 싶다. 근거 없는 BEST 10은 신뢰할 수 없기 때문이다.
4. 직장인으로서, **이동시간과 추천 일수**를 먼저 보고 싶다. 내 휴가 일수에 들어가는지가 1차 필터이기 때문이다.
5. 예산이 신경 쓰이는 사람으로서, **대략의 비용 레벨(¥~¥¥¥¥¥)**을 카드에서 바로 보고 싶다. 상세를 다 읽고 나서 포기하고 싶지 않기 때문이다.
6. 태풍·우기가 무서운 사람으로서, **리스크 경고(태풍철·혹서·혼잡)**를 추천과 함께 보고 싶다. 좋은 얘기만 듣고 갔다가 낭패 보기 싫기 때문이다.
7. 1년을 설계하는 사람으로서, **1~12월을 순서대로 훑으며** 연간 여행 지도를 그리고 싶다. 연차는 미리 계획해야 좋은 곳을 좋은 값에 가기 때문이다.
8. 재방문 사용자로서, **가고 싶은 곳을 위시리스트로** 모아두고 싶다. 볼 때마다 처음부터 뒤지기 싫기 때문이다. (P1)

### 2.6 성공 지표 (개인용 기준)
**정량**
- 실제 여행 계획 시 **첫 참조 도구**로 사용(분기 1회 이상 실여행에 기여).
- 시드 72곳 확보, 이후 월 2~3곳씩 추가(월별 후보 풀 확대).
- 시즌 근거 오류(시기·이벤트 착오) 신고 0건 유지 — 발견 즉시 수정, 연 1회 전수 감수.

**정성**
- "이번 달 어디 가지?" 검색에 쓰던 시간이 체감상 사라졌는가.
- 연휴 계획에서 국내/해외 비교가 이 앱 하나로 끝났는가.
- 추천 이유를 읽고 "아, 그래서 이 달이구나"라는 납득이 있었는가.

### 2.7 로드맵
- **Phase 1 — MVP (첫 출시):** P0 전부 + 시드 72곳. 목표: 다음 연휴 계획에 실제로 쓰기 시작.
- **Phase 2 — 실사용 편의:** 테마 필터·검색, 위시리스트·가봤음, 조건으로 찾기(월×일수×테마), 연휴 캘린더 오버레이. 여행지 100곳+로 확대(월별 후보 풀 4~5곳).
- **Phase 3 — 완성도:** 준비물 체크리스트, 한/일 토글, PWA, 사진·지도. 목표: 오프라인에서도 보는 "내 전용 여행 연감".

|---|---|---|

**(d) 이유 작성 원칙**
- "왜 이 달인가"는 반드시 위 (a)~(c)의 **구체적 사실**로 서술한다. ✗ "경치가 아름답다" → ✓ "수빙이 2월에 최대로 성장한다".
- 시기가 해마다 변하는 이벤트는 "**예년 ○월 ○순**"으로 표기하고, 고정 일정(네부타 8/2~7 등)만 날짜를 확정 표기.
- 리스크(태풍철·혹서·우기 걸침·성수기 혼잡)는 반드시 `cautions`에 함께 기록 — 좋은 얘기만 쓰지 않는다.

**시드 통계**: 국내 36곳(홋카이도 6·도호쿠 4·간토 7·주부 7·간사이 4·주고쿠 1·규슈 1·오키나와 5·기타 1), 해외 36곳(아시아 근거리~장거리·유럽·아프리카·오세아니아·아메리카 배분). 일부 여행지(후라노, 오키나와 본섬 등)는 복수 월에 등장 — 데이터는 1객체에 `monthlyReasons`로 통합하고 `calendar.json`에서 월별 참조.

### 3.5 콘텐츠 작성·운영 가이드

- **문체**: 카드·상세 모두 간결체(~다/명사형). 광고 문구·과장("꼭 가야 할", "인생 여행지") 금지.
- **이유의 검증 가능성**: `monthlyReasons`는 §3.1 프레임의 사실로만 구성. 감상은 `summary`·`highlights`로.
- **날짜 표기 원칙**: 매년 변동하는 행사는 "예년 ○월 ○순"으로. 고정 일정(네부타 8/2~7, 나담 7/11~13, 송크란 4/13~15 등)만 날짜 확정 표기.
- **연 1회 전수 감수(매년 1월)**: 이벤트 일정·요금 레벨·운행 정보(구로베 토롯코 등)를 공식 사이트 기준으로 점검. 게시 전에도 해당 연도 일정 확인.
- **리스크 고지 의무**: 태풍철(8~9월 오키나와·규슈), 우기 걸침, 혹서, 성수기 혼잡·고가는 `cautions`에 반드시 기록.
- **확장 규칙**: 새 여행지 추가 시 ① 스키마 준수 ② `validate:data` 통과 ③ 해당 월 `calendar.json` 후보 풀 검토 순서로.

## 5. 기술 설계 & 아키텍처

### 5.1 기술 스택 선정 & 근거

food-recipy에서 검증 완료된 구조를 그대로 재사용한다. 재검토 결과 여행 도메인에서도 결론이 달라질 이유가 없다.

| 선택지 | 판단 |
|---|---|
| **바닐라 JS(ES 모듈) ✅** | 빌드 0, 의존성 0, GitHub Pages 직배포. 이 앱의 복잡성은 콘텐츠이지 UI 로직이 아님. food-recipy에서 동일 규모(70+ 항목) 검증 완료 |
| React/Vue + 빌드 | 카드 리스트+상세 수준에 과투자. 빌드 체인 유지비만 증가 |
| SSG(Astro·11ty) | 정적 생성 매력은 있으나 "코드 수정 없이 JSON append"라는 운영 모델과 충돌 |
| Notion·스프레드시트 | 국내/해외×월 매트릭스 UI, 모바일 UX, URL 공유를 만들 수 없음 |

핵심 규칙(재사용): **상대경로 + 해시 라우팅**(Pages 하위 경로 대응), **`.nojekyll` 필수**, 한글 데이터 파일명은 `encodeURIComponent`로 fetch.

### 5.3 모듈 구조 & 의존 방향

```
app.js → router.js → views.js → data.js / calendar.js → config.js
                                   ↘ store.js (P1)
```

- `data.js`: 최초 진입 시 `calendar.json` + 두 destinations 파일을 **병렬 fetch**(총 ~수십 KB 수준이라 분할 lazy 불필요, 단순함 우선). 로드 후 `Map` 인덱스 구성. 실패 시 뷰에 에러 상태 전달.
- `calendar.js`: 순수 함수 중심 — `currentMonth(now)`, `nextMonth(m)`(12→1 순환), `picksFor(m)`(calendar.json→여행지 객체 배열), `isPeak(dest, m)`. **`Date`는 인자로 주입**해 테스트 가능하게.
- `views.js`: 문자열 템플릿 → `innerHTML` 렌더(단순), 사용자 입력 없는 정적 데이터라 XSS 표면 최소지만 텍스트는 이스케이프 유틸 통과.
- URL이 앱 상태의 전부(선택 월·구분 세그먼트는 쿼리 `#/month/9?scope=overseas`) — 새로고침·공유에 안전.

### 5.4 배포 파이프라인 (GitHub Pages)

1. `main` push → Settings → Pages → **Deploy from a branch** (`main` / root).
2. `.nojekyll` 커밋(한글 파일명·특수 경로 이슈 방지 — food-recipy에서 확인된 필수 사항).
3. 배포 URL: `https://taetaeng7-rgb.github.io/tripbook/` — 상대경로+해시 라우팅이라 하위 경로에서 그대로 동작.
4. 로컬 확인: `python3 -m http.server 8765` (ES 모듈+fetch라 `file://` 불가).

### 6.1 품질 리스크

### 6.2 데이터 검증 규칙 (`npm run validate:data`)
